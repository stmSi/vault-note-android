# VaultNote attachment encryption format

## Scope and versioning

Encryption format `1` protects imported attachment bytes and generated thumbnails at rest. It uses AES-256-GCM with a 128-bit authentication tag and a distinct nonce for every encryption operation. The key version is independent of the file-format version so keys can rotate without changing the parser.

Format `0` identifies legacy Phase 2 plaintext in app-private storage. It is accepted only by the bounded migration path and is never exposed to a viewer. Unsupported versions fail closed.

## Key storage

The current encryption key is a non-exportable 256-bit AES key generated by Android Keystore under:

```text
vaultnote.attachment.aes.v<key-version>
```

Version `1` is current. Key bytes are not placed in Room, preferences, files, logs, source, backups, or intent extras. Old aliases must remain available while any envelope references their version. A future rotation changes the current key version, retains historical keys for reads, and rewrites files independently before retiring an old alias.

The key is intentionally not configured with per-operation user authentication. Biometric or device-credential authentication gates the in-process vault session; Keystore protects the at-rest key. This prevents biometric enrollment changes from deliberately destroying attachment access, but it means code executing inside the compromised app process on an unlocked device may use the key.

## Binary envelope

All integers are unsigned concepts encoded in network byte order (big endian). Java signed storage is rejected when it produces an invalid range.

| Offset | Size | Field | Validation |
| ---: | ---: | --- | --- |
| 0 | 4 | Magic | ASCII `VNE1` |
| 4 | 1 | Format version | `0x01` |
| 5 | 4 | Key version | positive integer |
| 9 | 1 | Nonce length | exactly 12 |
| 10 | 8 | Plaintext length | 0 through 100 MiB |
| 18 | 12 | Nonce | provider-generated, unique per encryption |
| 30 | N | Ciphertext | exactly the declared plaintext length |
| 30 + N | 16 | GCM tag | verified before plaintext is released |

The complete file length must be exactly `30 + plaintext_length + 16`. Trailing data, truncation, invalid lengths, unknown versions, missing keys, and authentication failures are rejected.

## Authenticated context

GCM additional authenticated data is the complete 30-byte header followed by:

| Size | Field |
| ---: | --- |
| 4 | ASCII `VNC1` |
| 1 | purpose: `0x01` attachment or `0x02` thumbnail |
| 2 | UTF-8 record-ID length |
| variable | attachment record ID |

Record IDs are restricted to 1–128 ASCII letters, digits, `_`, and `-`. Binding the purpose and record ID prevents a valid ciphertext from being substituted for another attachment or reused as its thumbnail. The header authentication covers the file-format version, key version, nonce, and declared length.

## Write protocol

1. Copy and validate the untrusted source into a confined pending plaintext file while calculating SHA-256.
2. Ask the AES-GCM provider to initialize encryption and generate the nonce; callers cannot supply a nonce.
3. Stream plaintext through the cipher into a uniquely named pending envelope using 64 KiB buffers.
4. Write the tag, flush, and `fsync` the pending envelope.
5. Atomically move the envelope to its final app-private path on the same filesystem. If atomic move is unavailable, fail without replacing the prior file.
6. Delete the plaintext pending file in `finally` and persist format version `1` only after both file and Room operations succeed.

Interrupted imports are covered by the existing cleanup journal and `.pending-*` cleanup. Pending plaintext is confined to the app sandbox, never shared, and removed on normal completion; abandoned pending files are removed after the 24-hour safety window so cleanup cannot race an active import.

## Read protocol

Decryption uses one open read-only file descriptor and two streaming passes:

1. Parse and validate the exact envelope.
2. Resolve the key version.
3. Decrypt the entire ciphertext to a discard sink and verify the GCM tag.
4. Seek the same descriptor back to the ciphertext and decrypt again to the caller's pipe.

No plaintext is written to the caller before authentication succeeds. The second pass avoids a time-of-check/time-of-use path swap while keeping memory bounded. VaultNote does not create a plaintext viewer file; its private content provider streams verified bytes through a pipe.

## Legacy migration

After the first unlocked UI is available, migration processes at most eight legacy rows at a time. Before encrypting a raw original, it recomputes SHA-256 and compares it with the checksum already stored in Room. A mismatch is corruption and leaves the database version unchanged. Thumbnails have no historical checksum but remain bound to their attachment ID and purpose by GCM.

The migration is resumable after process death: if a file already has a format-1 envelope, the service authenticates it instead of encrypting it again. Room changes to version `1` only after the original and optional thumbnail both verify. A partially completed pair is therefore safe to retry.

## Failure behavior

- Authentication, key loss, malformed envelopes, context mismatch, and unsupported versions produce typed decryption failures and no plaintext output.
- Insufficient space and unavailable atomic replacement preserve the existing file and keep Room metadata unchanged.
- Cancellation is propagated and pending output is deleted.
- The original SHA-256 remains the content checksum; it is not a substitute for GCM authentication.
- Keystore loss, app-data clearing, or some device/OS failures can make encrypted attachments unrecoverable. Phase 6 encrypted backup is required for independent recovery.
